# 跨平台无障碍游戏设计与交互指南

> 本文档面向产品经理和交互/视觉设计师，重点阐述跨平台游戏（特别是引擎驱动、Canvas/WebGL 渲染的游戏）在无障碍设计上的核心理念、交互模式与语义组织方法。

---

## 一、无障碍游戏 vs 普通应用的核心区别

在设计无障碍游戏时，首先要理解它与普通应用在信息传递上的本质差异：

| 维度 | 普通应用 | 无障碍游戏 |
|------|---------|-----------|
| **信息密度** | 一屏可展示大量信息 | 一次只传递一个焦点元素的信息，必须精炼 |
| **操作节奏** | 用户自主控制节奏 | 游戏有外部节奏（轮次、倒计时），需主动通知状态变化 |
| **信息类型** | 文本为主，结构清晰 | 游戏状态是动态、隐式的，需主动播报转换 |
| **空间布局** | 视觉布局引导操作 | 焦点顺序即信息架构，顺序比位置重要 |
| **时间压力** | 通常无 | 游戏有轮次等待、AI行动，需要状态变化时的主动播报 |
| **错误容忍** | 撤销即可 | 盲人无法"看"到误操作后果，需更强的防误触机制 |
| **信息组织** | 并行展示（一眼看到全部） | 串行展示（逐个焦点浏览），需要全局摘要入口 |

**核心差异总结**：普通应用是"用户主动获取信息"，无障碍游戏是"系统必须主动推送关键状态变化"。

## 二、从现有实践中总结的关键经验

### 1. 不要假设系统无障碍会自动覆盖主玩法

对列表、按钮、开关、输入框这类标准 UI，系统无障碍通常非常可靠。
但对棋盘、手牌、战场、时间轴、组合动作、结果摘要等游戏语义，系统并不会天然理解。

更稳妥的判断是：
> 这类游戏并不是“只要给控件补标签，系统读屏就能玩”，而是需要游戏层主动建立语义模型和播报策略。

### 2. 系统读屏是基础设施，不是完整方案

对于棋类、牌类、策略类、战斗回合类游戏，真正可用的无障碍体验通常来自“游戏自己维护的一层可访问语义系统”。
换句话说，跨平台无障碍游戏不应该只问“这个平台有没有 VoiceOver / TalkBack”，而应该先问：
- 游戏有没有把视觉状态压缩成可听懂、可导航、可执行的语义节点？

### 3. 系统能力只能解决“标准 UI”，解决不了“游戏语义”

系统读屏非常擅长：朗读标准按钮、管理基础焦点移动、暴露标准动作。
但系统读屏并不天然理解：一盘棋当前谁占优、一手牌里哪些信息真正重要、一个棋子有哪些合法走法。
这些都必须由游戏自己建模。

### 4. 无障碍游戏的核心不是“把视觉原样念出来”

正确目标不是“把屏幕上所有东西都播报一遍”，而是把用户完成任务需要的信息抽出来：
- 当前是谁的回合
- 你有哪些可执行动作
- 每个动作执行后的结果是什么
- 是否需要确认
这要求游戏自己做“语义压缩”，而不是仅仅堆平台属性。

## 三、适用于跨平台无障碍游戏的交互模式

### 1. 完全兼容系统读屏，不自定义手势

✅ 正确：使用 VoiceOver/TalkBack 原生手势（左/右滑动切换焦点、双击确认）。
❌ 错误：自定义左右滑动为"选牌"、上下滑动为"切换"，与读屏手势冲突。
> 如果需要用户关闭读屏才能玩，那不是无障碍，是重新造了一个障碍。

### 2. 焦点顺序优先于视觉排版（焦点顺序即信息架构）

推荐统一模式：
`状态摘要 -> 当前可浏览对象 -> 当前可执行动作 -> 辅助信息`
这比“按屏幕从左到右、从上到下”更符合无障碍用户的任务路径。不要按视觉布局排列焦点，导致盲人需要反复跳转才能理解上下文。

### 3. 提供摘要入口，而不是逼用户扫完整屏

每个高密度界面都应该有摘要能力，例如：棋盘摘要、手牌摘要、战场摘要、资源摘要。
用户应能在 1 到 2 次操作内获得“现在发生了什么”的全局判断。串行浏览的效率远低于并行视觉，必须有"一键概览"来弥补。

### 4. 把复杂选择拆成“浏览 -> 选择 -> 确认”（选择与确认分离）

典型模式：先浏览对象 -> 再进入动作选择 -> 最后确认执行。
这样可以减少误触，也更适合读屏场景。触摸是"探索"，双击才是"执行"。两者分离，防止误操作。

### 5. 不把精确位置作为核心输入

应避免：拖到某个像素位置、长按某个隐藏热区、依赖视觉路径手势。
应优先：焦点切换、动作列表、语义分区、结构化命令。

### 6. 主动播报“状态变化”，而不是等待用户自己发现

必须主动提示的事件包括：
- 轮到玩家操作（回合轮转）
- 其他玩家/系统行动结果
- 游戏阶段切换
- 操作成功、失败或非法操作反馈
- 倒计时临界点
- 页面切换完成

### 7. 音频反馈分层

| 层级 | 用途 | 示例 |
|------|------|------|
| 语音 | 完整信息 | "红桃3，已选中" |
| 音效 | 事件标识 | 选中叮声、错误嗡声 |
| 震动 | 操作确认 | 选中短震、非法双震 |

> 三层反馈各有分工，不可互相替代。没有语音的音效是噪音，没有音效的语音缺少节奏感。

### 8. 页面跳转与返回按钮的焦点规范（“返回按钮陷阱”）

在移动端和跨平台 UI 中，左上角的“返回”按钮往往是视觉上的第一个元素。如果不对页面切换做特殊处理，屏幕阅读器默认会把焦点落在“返回”上并仅仅朗读“返回，按钮”。这会导致视障用户在页面跳转后完全不知道自己进入了哪个页面。

**推荐的通用规范：**
1. **主动播报页面名称（首选）**：页面加载完成时，触发一次系统级的无障碍通知，主动朗读当前页面名称。随后焦点自然落在“返回”按钮上。
2. **增加隐藏的页面标题焦点（备选）**：在“返回”按钮的 DOM/节点树之前，插入一个不可见（或顶部居中可见）的文本节点作为第一个焦点，无障碍角色设为“标题”。

**返回按钮的自身属性要求：**
- **无障碍名称 (Label)**：必须是 `返回`（严禁使用“左箭头”、“icon_back”等）。
- **角色 (Role)**：`按钮`。
- **无障碍提示 (Hint)**：建议补充上下文，如 `返回主界面` 或 `返回上一页`。

### 9. 触控最大化与状态听觉化（Audio-First & Touch-Maximized）

在为视障或精细运动障碍用户设计 UI 时，**“看得见”不等于“好操作”**。健全人习惯的“所见即所得”排版，往往会为了展示信息而牺牲掉宝贵的触控面积。

**核心设计原则：**
1. **触控区域最大化（Touch-Maximized）**：极其重要的操作（如增加/减少局时），不要为了在中间挤出空间显示“当前数值”而把按钮做小。巨大的色块或全宽按钮是盲操的最佳形态。焦点区域必须 ≥44×44pt。
2. **状态听觉化（Audio-First）**：界面上的当前数值或当前状态，**无需强行占用视觉排版空间**。只要焦点落在操作按钮上，读屏软件能通过 `Value` 属性读出当前状态，且点击后能立刻通过语音播报给予反馈，这就足够了。
3. **二元切换的单按钮化**：对于红/黑方、开/关等二元状态切换，**禁止使用占用空间的“分段控制器(Segmented Control)”或“单选框组”**。应使用单一的巨大按钮，通过点击直接切换状态，并完全依赖语音播报切换结果。

## 四、游戏语义压缩的具体建议

跨平台真正可复用的，不是平台 API，而是“如何把视觉复杂度压缩成语言复杂度”。

### 1. 棋牌游戏
应优先表达：谁的回合、棋子名称、当前方位或所属区域、可执行动作、走后结果。
不应优先表达：像素坐标、纯颜色变化、用户不需要决策的装饰信息。

### 2. 牌类游戏
应优先表达：点数、数量、牌型、可出/不可出、当前轮到谁。
如果花色不是决策关键，就不要强行播报花色。

### 3. 资源经营与策略类
应优先表达：当前资源摘要、缺口与风险、当前可执行高价值动作、行动结果。
不应让用户在一长串建筑和数值中自己筛重点。

### 4. 聊天、社交、战报
应优先表达：发送者、类型、是否未读、关键内容摘要。
例如“系统消息，奖励已发放，未读”比把完整富文本原样读出来更有用。

## 五、设计检查清单 (Checklist)

在每次设计评审时，逐项检查：

- [ ] 所有交互是否兼容系统读屏（无需关闭VoiceOver/TalkBack）？
- [ ] 焦点顺序是否符合认知逻辑（状态→内容→操作）？
- [ ] 语音播报是否只包含决策所需信息（无视觉信息冗余翻译）？
- [ ] 状态变化是否有主动语音/震动通知？
- [ ] 是否提供了全局摘要/概览入口？
- [ ] 是否有防误触机制（选择与确认分离）？
- [ ] 操作是否可撤销/可修正？
- [ ] 是否避免了精确触摸位置依赖？
- [ ] 等待状态是否有语音提示？
- [ ] 是否仅靠颜色区分状态？
- [ ] 时间限制是否有延长/关闭选项？
- [ ] 操作反馈是否包含充分上下文？
- [ ] 信息流速度是否可控（可暂停/重听）？
- [ ] 焦点区域是否足够大（≥44×44pt）？
- [ ] 音效是否仅作辅助（不独立承载关键信息）？
