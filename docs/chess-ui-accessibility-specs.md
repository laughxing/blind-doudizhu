# 象棋无障碍 UI 设计与开发规格 (A11y UI Specs)

> **文档定位**：本文档是产品、交互、视觉、客户端和测试团队的共同契约。
> **目标**：把当前象棋产品的关键路径从“能被读出来”提升到“能被独立理解、独立操作、独立验收”。
> **适用范围**：本文档覆盖当前已出视觉稿中的 `主界面`、`下棋模块`、`好友对战设置`、`人机对战设置`、`对局页`、`退出确认弹窗`。
> **暂不展开页面**：`旁观`、`个人信息`、`设置`、`操作指引`、`棋谱详情`、`结算页` 等未出稿或未定稿页面，后续按本文同样结构补充。

## 1. 页面总览与流转

```text
主界面
  -> 下棋模块
    -> 好友对战设置
      -> 高级计时设置（可选）
      -> 创建房间 / 邀请好友
        -> 对局页
          -> 退出确认弹窗
    -> 人机对战设置
      -> 开始挑战
        -> 对局页
          -> 退出确认弹窗
```

| 流转 | 触发条件 | 进入后必须让用户立即知道的信息 |
| --- | --- | --- |
| 主界面 -> 下棋模块 | 点击 `下棋` | 已进入下棋模块；可选择哪些模式 |
| 下棋模块 -> 好友对战设置 | 点击 `好友对战` | 当前是好友对战；能否直接开始还是需要先创建/邀请 |
| 下棋模块 -> 人机对战设置 | 点击 `人机对战` | 当前难度、当前执棋方、如何开始 |
| 好友对战设置 -> 高级计时设置 | 点击 `计时规则` 或参数项 | 当前计时模式、当前局时/步时/读秒 |
| 好友对战设置 -> 对局页 | 创建成功且房间已进入对局 | 你执红/执黑、轮到谁、上一步/初始局面 |
| 人机对战设置 -> 对局页 | 点击 `开始挑战` 且创建成功 | 你执红/执黑、AI 难度、轮到谁 |
| 对局页 -> 退出确认弹窗 | 点击 `退出` | 这是确认对话框；默认安全选项是什么 |
| 退出确认弹窗 -> 对局页 | 点击 `否` 或系统返回 | 已返回对局；焦点恢复到 `退出` 按钮 |

## 2. 全局无障碍设计原则

### 2.1 只使用系统标准手势

- 读屏浏览只依赖 `左/右滑动切换焦点`、`双击激活`、`系统返回手势`。
- 禁止把自定义左右滑、长按滑动路径、拖拽落点设计成主路径操作。
- 如保留 `麦克风` 语音入口，它只能是**可选增强能力**，不能成为唯一操作方式。

### 2.2 焦点顺序就是信息架构

- 高密度页面统一采用：`页面标题/状态摘要 -> 当前可浏览对象 -> 当前可执行动作 -> 辅助信息`。
- 页面进入后必须先让用户知道“我在哪”“现在是什么状态”“下一步能做什么”。
- 返回时焦点必须恢复到触发返回的元素，避免用户重新扫屏。

### 2.3 每个复杂页面都必须有摘要入口

- 不能要求用户靠多次滑动自己拼出上下文。
- 至少以下页面必须有摘要节点或主动播报：
  - `下棋模块`
  - `好友对战设置`
  - `人机对战设置`
  - `对局页`
- 摘要应在 1 到 2 次操作内可重听，不得只能在首次进入时听一次。

### 2.4 选择与确认分离

- 对局页主路径必须是：`浏览棋子 -> 选择棋子 -> 浏览合法着法 -> 确认着法`。
- `退出`、`认输` 等不可逆操作必须二次确认。
- `悔棋`、`提和`、`邀请好友`、`创建房间` 失败后要有明确的可恢复路径。

### 2.5 动态状态必须主动播报

- 必播报事件包括：
  - 轮到谁走
  - 对手走子结果
  - 将军 / 被将军
  - 非法操作
  - 倒计时临界点
  - 房间创建成功/失败
  - 好友加入 / 对手离开
  - 网络中断 / 重连成功
  - 页面切换完成

### 2.6 反馈分层

| 通道 | 用途 | 示例 |
| --- | --- | --- |
| 语音 | 完整语义 | `轮到你走。上一步黑马 8 进 7。` |
| 音效 | 事件节奏 | 轮到我方提示音、非法操作嗡声、将军警示音 |
| 震动 | 操作确认 | 选中短震、错误双震、胜负长震 |

> 三层反馈不可相互替代。音效和震动只能增强节奏，不能代替关键语义。

### 2.7 视觉与触控约束

- 文本与背景对比度不低于 `4.5:1`。
- 所有可交互元素热区不小于 `44x44 dp/pt`。
- 不可仅靠颜色表达状态，选中态需配合描边、字重、图标或底纹。
- 系统字体放大到 `200%` 时，按钮高度和容器必须可拉伸，不能截断。
- 获焦态必须明确可见，尤其是棋子、操作按钮和弹窗默认按钮。

## 3. 对局页统一语义模型

### 3.1 不把 90 个交叉点当成默认焦点流

- `棋盘 9x10` 的所有点位不能直接线性暴露为默认读屏顺序，否则浏览成本过高。
- 对局页主路径必须是语义化节点，而不是像素化节点。
- 默认浏览对象应是：
  - `局面摘要`
  - `当前可操作棋子列表`
  - `当前棋子的合法着法列表`
  - `全局动作`
- 如果产品后续支持“棋盘坐标探索”，它也只能作为**辅助模式**，不能替代主路径。

### 3.2 对局摘要必须包含的内容

- 当前执棋方：你执红 / 你执黑
- 当前轮次：轮到你 / 轮到对手
- 上一步：例如 `黑炮 2 平 5`
- 危险状态：`当前未将军` / `你被将军` / `你已将军`
- 时间信息：双方剩余局时，若启用读秒则补充读秒状态
- 当前选择：未选中棋子 / 已选中 `红车九`

### 3.3 棋子节点与着法节点的合同

- **棋子节点**需要提供：
  - `Label`：棋子名称，如 `红车九`
  - `Value`：当前位置、当前是否可动、合法着法数量
  - `Hint`：`双击查看可走位置`
- **着法节点**需要提供：
  - `Label`：动作名，如 `前进一格`
  - `Value`：落点、是否吃子、是否将军
  - `Hint`：`双击确认着法`

### 3.4 对局主路径时序

```text
进入对局
  -> 播报局面摘要
  -> 浏览可操作棋子
  -> 双击某棋子
  -> 播报该棋子的合法着法数
  -> 浏览合法着法
  -> 双击确认
  -> 播报走子结果
  -> 等待对手 / 轮到自己继续操作
```

### 3.5 防误触规则

- 选中棋子后必须出现 `取消选择`。
- 当前无合法着法的棋子不能作为主路径上的可执行节点。
- 非法走子必须立即反馈，不得“无声失败”。
- 当轮到对手时，出子相关节点应隐藏或标记为不可用，并播报 `等待对手走子`。

## 4. 页面规格

## 4.1 主界面

### 4.1.1 页面目标与信息层级

- **页面职责**：应用一级导航，引导用户进入下棋主路径或其他辅助功能。
- **核心路径**：打开应用 -> 进入 `下棋`。
- **信息层级**：
  1. `下棋`
  2. `旁观`、`个人信息`、`设置`、`操作指引`
  3. `切换账号`

### 4.1.2 页面切换与焦点流

- **进入播报**：`已进入主界面。可选择下棋、旁观、个人信息、设置、操作指引。`
- **默认焦点**：隐藏标题或摘要节点 `主界面`。
- **返回焦点恢复**：从子页面返回时，恢复到之前点击的入口按钮。
- **焦点顺序**：`主界面摘要 -> 下棋 -> 旁观 -> 个人信息 -> 设置 -> 操作指引 -> 切换账号`

## 4.4 人机对战设置

### 4.4.1 页面目标与信息层级

- **页面职责**：配置 AI 难度与执棋方并开始挑战。
- **核心路径**：确认难度 -> 确认执棋方 -> 点击 `开始挑战`。
- **信息层级**：
  1. `开始挑战`
  2. 当前难度
  3. 执棋方
  4. 返回

### 4.4.2 页面切换与焦点流

- **进入播报**：`已进入人机对战设置。当前难度 {difficulty}，当前 {side}，可调整后开始挑战。`
- **默认焦点**：摘要节点 `人机对战设置`。
- **返回焦点恢复**：从对局返回时恢复到 `开始挑战`。
- **焦点顺序**：`人机对战设置摘要 -> 当前难度 -> 降低难度 -> 增加难度 -> 执棋方 -> 开始挑战 -> 返回`

### 4.4.3 控件与状态合同

## 4.5 对局页（核心页面）

### 4.5.1 页面目标与信息层级

- **页面职责**：让用户在读屏环境下独立完成浏览局面、选择棋子、确认着法和执行全局操作。
- **核心路径**：进入对局 -> 听摘要 -> 选棋子 -> 选合法着法 -> 确认 -> 等待对手。
- **信息层级**：
  1. 当前轮次与危险状态
  2. 当前可操作棋子
  3. 当前棋子的合法着法
  4. 全局动作：悔棋、提和、认输、退出
  5. 辅助动作：重读摘要、轮到谁、历史浏览、麦克风（若保留）

### 4.5.2 视觉设计约束

- 棋盘可以保留视觉布局，但无障碍主路径不能依赖用户准确点中某个交叉点。
- 选中棋子、合法落点、最近一步必须有明显非颜色化视觉标识。
- 顶部辅助按钮与底部全局按钮必须有稳定顺序，不得动态换位。
- 当控件不可用时，需有禁用态视觉和无障碍状态，不能只隐藏提示。

### 4.5.3 页面切换与焦点流

- **进入播报**：`已进入对局。你执{side}。{turnInfo}。上一步：{lastMove}。{checkStatus}。`
- **默认焦点**：`局面摘要`。
- **返回焦点恢复**：
  - 从 `退出确认弹窗` 返回时，恢复到 `退出`。
  - 从系统中断返回时，恢复到上次获焦的语义节点。
- **焦点顺序**：
  - `局面摘要`
  - `重读摘要`
  - `当前可操作棋子列表`
  - `合法着法列表（仅在已选中棋子时出现）`
  - `取消选择`
  - `轮到谁`
  - `开始局面`、`最新局面`、`上一步`、`下一步`（如保留）
  - `悔棋`、`提和`、`认输`、`退出`
  - `麦克风（若保留）`

### 4.5.6 对局关键时序要求

1. **进入对局**：先播报摘要，再允许用户浏览棋子。
2. **轮到自己**：必须有提示音或短震，并播报 `轮到你走`。
3. **选中棋子**：立即播报该棋子及合法着法数量。
4. **确认着法**：必须播报完整着法结果，如 `红车九进一，吃卒，将军`。
5. **轮到对手**：隐藏或禁用出子路径，播报 `等待对手走子`。
6. **对手走子完成**：立即播报结果，并把焦点恢复到 `局面摘要`。
7. **非法操作**：必须用高优先级语音 + 错误音效或双震提示。

## 6. 验收清单

### 6.1 设计验收

- [ ] 所有关键路径均可在系统读屏开启状态下完成，无需关闭读屏。
- [ ] 页面进入后，用户能在 1 到 2 次操作内知道自己在哪个页面。
- [ ] 高密度页面均提供摘要入口，并可重复触发。
- [ ] 对局页默认不是 90 个点位线性浏览，而是语义化路径。
- [ ] 焦点顺序符合 `摘要 -> 对象 -> 动作 -> 辅助信息`。
- [ ] 所有状态变化都有主动反馈，不要求用户自己猜。
- [ ] `退出`、`认输` 等危险操作具备二次确认。
- [ ] 不可用状态、选中态、最近一步不只靠颜色表达。
- [ ] 文本放大到 `200%` 时无截断、无重叠。
- [ ] 热区满足 `44x44 dp/pt`。

### 6.2 对局专项验收

- [ ] 用户能在 `5 到 10 秒` 内知道现在轮到谁、自己能做什么。
- [ ] 用户无需依赖视觉坐标即可完成一整步走子。
- [ ] 选中棋子后，合法着法是可浏览、可确认、可取消的。
- [ ] 对手走子后，系统会主动播报结果并恢复到合理焦点。
- [ ] 非法操作不会静默失败。
- [ ] 倒计时临界点有提示，且不会播报轰炸。
- [ ] 系统读屏与游戏自定义播报不会重复读同一条信息。

### 6.3 测试场景

- [ ] 系统读屏开启与关闭分别测试。
- [ ] 慢语速与快语速分别测试。
- [ ] 人机对战与好友对战分别测试。
- [ ] 建房成功、建房失败、邀请失败分别测试。
- [ ] 对局中断线、后台切换、重连恢复分别测试。
- [ ] 弹窗打开后系统返回、点 `否`、点 `是` 分别测试。
