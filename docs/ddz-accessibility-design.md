# 盲人斗地主交互设计方案

> 基于 iOS VoiceOver / Android TalkBack 无障碍交互范式，为单机斗地主设计的完整交互方案。

---

## 一、页面总览与流转

```
首页 ──→ 叫地主页 ──→ 抢地主页 ──→ 出牌页 ──→ 结算页
 ↑                                                      │
 └──────────────── 重新开始 ←───────────────────────────┘
```

| 流转条件 | 说明 |
|---------|------|
| 首页 → 叫地主页 | 点击「开始游戏」 |
| 叫地主页 → 抢地主页 | 有人叫地主后，其余玩家可选择抢地主 |
| 叫地主页 → 出牌页 | 叫/抢地主确定后，底牌发放完成 |
| 出牌页 → 结算页 | 任一方手牌出完 |
| 结算页 → 首页 | 点击「重新开始」 |

---

## 二、核心设计思路

### 手势定义（仅用系统标准手势）

| 手势 | 含义 | 说明 |
|------|------|------|
| 左/右滑动 | 切换焦点 | 浏览手牌/按钮/信息，完全由读屏处理 |
| 双击 | 确认/进入 | 选中牌点进入选数量、确认出牌等 |
| 双指Z字滑动 | 返回 | 系统标准返回手势 |

**就这三种，零学习成本。**

### 手牌组织方式：按牌点分组

斗地主的牌只看点数，花色无意义。手牌按点数分组展示：

> 例：手牌有 3♠ 3♥ 5♣ 5♦ 5♠ 7♥ 7♦ K♠ K♥ K♣ K♦ A♥ 2♠ 小王 大王

读屏浏览顺序为：
- "3，2张"
- "5，3张"
- "7，2张"
- "K，4张"
- "A，1张"
- "2，1张"
- "小王"
- "大王"

这样17张牌只需要8次滑动就能浏览完，而非17次。

### 选牌逻辑：选点数+选数量

在某个牌点上双击进入选牌，再通过左右滑动选择出几张：

> 双击"5，3张" → "出1张5" / "出2张5：对5" / "出3张5：三条5"

每次切换都自动播报对应牌型名称，双击确认选择。

### 语音播报规则

| 场景 | 播报内容 |
|------|---------|
| 浏览手牌 | "3，2张" "5，3张" "大王" |
| 选中/取消牌 | "选中5，3张" "取消5" |
| 切换功能区 | "手牌区，共17张" "操作按钮区" |
| 他人出牌 | "上家出了：对5" "下家出了：单张9" |
| 轮到你 | 提示音 + "轮到你出牌" |

### 音效设计

| 音效 | 含义 |
|------|------|
| 叮 | 选中一张牌 |
| 嗡 | 取消选中 |
| 🔔提示音 | 轮到你操作 |
| 出牌音 | 成功出牌 |
| 炸弹音 | 炸弹/火箭出牌 |
| 胜利音 | 赢了 |
| 失败音 | 输了 |

## 三、各页面交互详设

### 页面1：首页

**功能**：游戏入口、模式选择、无障碍设置

**交互元素（按焦点顺序）**：

| 序号 | 元素 | 类型 | VoiceOver朗读 | 操作 |
|-----|------|------|-------------|------|
| 1 | 游戏标题 | 标题 | "斗地主，盲人无障碍版" | 纯展示 |
| 2 | 开始游戏 | 按钮 | "开始游戏，双击进入" | 双击进入叫地主页 |
| 3 | 难度选择 | 选项 | "难度：简单，上下滑动切换" | 上下滑动在 简单/中等/困难 间切换 |
| 4 | 无障碍设置 | 按钮 | "无障碍设置，双击进入" | 双击进入设置页 |

**交互时序**：

```
1. 页面加载 → 语音播报："欢迎来到斗地主，盲人无障碍版"
2. 焦点自动定位到「开始游戏」按钮
3. 用户双击「开始游戏」→ 播报"游戏开始" + 震动反馈
4. 自动发牌（动画隐藏）→ 播报"发牌完成，您手上有17张牌"
5. 跳转至叫地主页
```

---

### 页面2：叫地主页

**功能**：玩家决定是否叫地主

**交互元素**：

| 序号 | 元素 | 类型 | VoiceOver朗读 | 操作 |
|-----|------|------|-------------|------|
| 1 | 手牌概览 | 信息 | "您手上有17张牌，向右滑动查看" | 左右滑动按牌点浏览 |
| 2 | 单牌点 | 卡片 | "3，2张" / "K，4张" | 双击进入选数量 |
| 3 | 叫地主 | 按钮 | "叫地主，双击确认" | 双击叫地主 |
| 4 | 不叫 | 按钮 | "不叫，双击确认" | 双击不叫 |

**交互时序**：

```
1. 进入页面 → 播报："叫地主阶段，请选择是否叫地主"
2. 焦点定位到「叫地主」按钮
3. 用户可先左右滑动手牌查看（播报牌点和数量）
4. AI先手叫地主场景：
   a. AI叫地主 → 播报"上家叫了地主"
   b. 轮到玩家 → 播报"轮到您，是否叫地主？"
5. 用户双击「叫地主」→ 播报"您叫了地主" + 震动
6. 用户双击「不叫」→ 播报"您选择不叫"
7. 若三人都不叫 → 播报"无人叫地主，重新发牌" → 回到发牌
8. 叫地主完成 → 判断是否进入抢地主阶段
```

---

### 页面3：抢地主页

**功能**：其余玩家决定是否抢地主

**交互元素**：

| 序号 | 元素 | 类型 | VoiceOver朗读 | 操作 |
|-----|------|------|-------------|------|
| 1 | 叫地主信息 | 信息 | "当前地主候选：上家，底牌3张" | 纯展示 |
| 2 | 抢地主 | 按钮 | "抢地主，双击确认" | 双击抢地主 |
| 3 | 不抢 | 按钮 | "不抢，双击确认" | 双击不抢 |

**交互时序**：

```
1. 进入页面 → 播报："抢地主阶段"
2. 若AI先叫地主 → 播报"上家叫了地主，是否抢地主？"
3. 若玩家叫地主 → 等待AI是否抢 → 播报"下家抢了地主/无人抢地主"
4. 用户双击「抢地主」→ 播报"您抢了地主" + 震动
5. 用户双击「不抢」→ 播报"您选择不抢"
6. 抢地主结束 → 播报"地主确定为XXX，底牌为：大王、小王、2"
7. 焦点切换到手牌 → 播报"底牌已加入，您现在有20张牌"
8. 跳转至出牌页
```

---

### 页面4：出牌页（核心页面）

**功能**：选牌、出牌、提示牌型、查看场上信息

**三个焦点区域，双指左右切换**（读屏的转子/区域切换功能）：

#### ① 手牌区

- 左右滑动浏览牌点："3，2张""5，3张"…
- 双击某牌点 → 进入数量选择："出1张5" / "出2张5：对5" / "出3张5：三条5"
- 双击确认 → 该牌标记为"已选"
- 已选的牌点会变为："5，3张，已选3张"

#### ② 操作按钮区

| 按钮 | 条件 | 语音 |
|------|------|------|
| 出牌 | 有合法牌型 | "出牌，双击确认" |
| 不出 | 上家出过牌 | "不出，双击跳过" |
| 提示 | 轮到自己 | "提示，双击查看可出牌型" |
| 清空选择 | 有已选牌 | "清空选择，双击取消所有已选" |

#### ③ 战况信息区

| 元素 | 语音 |
|------|------|
| 角色信息 | "你是地主/农民" |
| 上家信息 | "上家：农民，剩8张" |
| 下家信息 | "下家：农民，剩12张" |
| 上次出牌 | "上家出了：对5" |

**提示功能交互**：

```
1. 用户双击「提示」→ 系统分析当前可出牌型
2. 若有可出牌型 → 播报"可出：对K" + 自动选中对应牌 + 震动
3. 再次双击「提示」→ 切换到下一个可出牌型，播报"可出：三带一对，3三个带5一对"
4. 若无可出牌型 → 播报"当前没有可出的牌型，请选择不出"
5. 提示循环：所有牌型提示完后回到第一个
```

**交互时序（自由出牌 — 地主首出）**：

```
1. 进入出牌页 → 播报"您是地主，请自由出牌"
2. 焦点定位到手牌区
3. 用户左右滑动浏览手牌，双击选中要出的牌
4. 选中完成后，右滑到「出牌」→ 双击确认
5. 系统校验牌型：
   a. 合法 → 播报"出牌成功：顺子3到7" + 震动 → 等待AI出牌
   b. 不合法 → 播报"牌型不合法，请重新选牌" + 双震动 → 焦点回手牌区
6. AI出牌 → 播报"上家出了：单张10" → 焦点回手牌区
7. 循环直到一方出完
```

**交互时序（跟牌）**：

```
1. 播报"上家出了：对5，请跟牌"
2. 焦点定位到「提示」按钮（引导用户先看提示）
3. 用户双击「提示」→ 播报可出牌型并自动选牌
4. 用户双击「出牌」确认 / 双击「不出」跳过
5. AI继续出牌 → 循环
```

**交互时序（AI轮次 — 等待状态）**：

```
1. 播报"等待上家出牌..."
2. AI思考（1-2秒延迟模拟）
3. AI出牌 → 播报"下家出了：飞机带翅膀，333444带56"
4. 轮到玩家 → 播报"轮到您" + 震动提醒
5. 焦点自动移至「提示」按钮
```

```
4. 用户选择重新开始或再来一局
```

## 四、核心时序总结

```
首页 ──→ 发牌(播报"发牌完成，17张牌")
  ──→ 叫地主(播报"是否叫地主？")
    ──→ 抢地主(播报"是否抢地主？")
      ──→ 确定地主(播报"地主确定，底牌XXX")
        ──→ 出牌循环 {
             地主出牌 → AI1出牌 → AI2出牌 →
             跟牌/不出 → ...
           }
          ──→ 某方出完 → 结算(播报胜负)
            ──→ 重新开始/再来一局
```

## 五、关键改进总结

1. **零手势冲突**：所有操作基于 VoiceOver/TalkBack 标准交互，用户无需关闭读屏
2. **不报花色**：只播报点数和数量，信息精炼
3. **按牌点分组**：浏览效率大幅提升（8次滑动 vs 17次）
4. **选牌=选点数+选数量**：符合斗地主的出牌逻辑，而非模拟逐张选牌
5. **三个手势搞定一切**：左/右滑动、双击、双指Z字返回，零学习成本
6. **主动播报状态**：轮次切换、AI出牌、游戏阶段变化均主动通知
7. **提示即引导**：跟牌时焦点优先定位到提示按钮，降低认知负荷

## 六、从《天天象棋（无障碍版）》APK 反推其无障碍交互实现

> 这一节不是产品方案，而是基于 APK 静态资源做的实现反推。当前证据主要来自 `assets/*.html` 和 `assets/res/import/*.json` 中的 `cc.Prefab` 资源；**尚未拿到 `.jsc` 业务源码还原，也没有在资源层直接看到大量 Android 原生 accessibility API 调用**，因此这里的结论以“交互结构证据”为主，避免过度推断。

### 1. 总体判断

这版“无障碍版”并不是单纯依赖原生 `contentDescription`、`TalkBack` 或 `announceForAccessibility` 之类关键词来显式实现无障碍，而更像是通过以下方式降低门槛：

1. **把首次进入流程讲清楚**：先协议、再权限说明、再进入业务。
2. **把主路径做短**：首页和功能页尽量采用标题 + 列表入口 + 少量主按钮。
3. **把复杂操作离散化**：语音下棋、局面编辑、棋谱编辑都被拆成可逐项确认的固定按钮或槽位。
4. **把系统状态文字化**：加载中、暂无数据、轻触重试、返回、输入棋谱 ID 等文案都直接暴露在资源里。
5. **把传播链路收敛**：二维码、固定分享目标、保存图片等都被做成明确入口，而不是依赖用户自己摸索系统菜单。

换句话说，这个 APK 的无障碍实现更像是**产品级交互约束**，而不是单点的系统无障碍属性补丁。

### 2. 首次进入：协议与权限前置，降低理解门槛

#### 2.1 协议同意链路

`assets/msdk_policy_content.html` 明确写到：

- 使用前先阅读 `游戏许可及服务协议`、`天天象棋隐私保护指引`、`游戏儿童隐私保护指引`、`第三方信息共享清单`
- 最后以“**请点击 ‘同意’ 开始使用我们的服务**”收束

这说明首次进入并不是把用户直接丢进登录或首页，而是采用了**阅读 → 同意 → 继续**的线性路径。对无障碍用户来说，这种顺序比多弹窗叠加、临时插入授权框更容易理解，也更适合读屏逐项浏览。

#### 2.2 权限解释先于权限弹窗

`assets/msdk_permission_content.html` 中的文案非常关键：

- `存储权限`：`保存图片并体验【游戏分享】和【资源下载更新】等游戏功能`
- `麦克风权限`：`游戏内语音下棋功能`

这说明它并不是“直接申请权限”，而是先解释**为什么要这个权限**。尤其是“分享”和“语音下棋”都被直接写在权限说明里，意味着这些能力不是边缘功能，而是预期主路径的一部分。

### 3. 首页与功能入口：用规则化列表替代复杂自由布局

#### 3.1 首页壳层很轻

`assets/res/import/0d/0d0b18fde.597e6.json` 中可见：

- 标题为 `首页`
- `_N$showBackButton:false`
- `_N$showHelpButton:false`

这类启动壳页面的信号很明确：**先给用户一个简单、稳定、无干扰的顶层入口**，而不是在首页同时堆很多悬浮入口和复杂控件。

#### 3.2 学棋页是标准“标题 + 列表项”结构

`assets/res/import/0c/0c0377726.197d2.json` 中可见：

- 页面标题：`学棋`
- 入口项：`启蒙课程`、`在线课程`、`题库训练`、`自我分析`、`大师开局`、`棋谱&记谱`

这说明首页至少有一部分能力入口采用了**统一样式的条目式导航**。对无障碍来说，这种结构的好处是：

- 焦点顺序天然稳定
- 每一项都是“一个名字 + 一个动作”
- 比自由排布卡片、混合横滑区域更容易建立心智模型

### 4. 棋谱使用路径：搜索、ID 直达、空状态提示都被显式做出来

#### 4.1 搜索框不是隐藏能力，而是正面入口

`assets/res/import/0e/0e1f815fc.53516.json` 中可见：

- 搜索占位文案：`棋谱名称/棋手`
- 返回事件：`handler:"onReturnEvent"`
- 状态文案：`请稍后轻触重试`、`暂无数据`

这说明棋谱库并不是纯浏览页，而是明确支持**按名称/棋手搜索**，同时把加载失败和空结果都做成了可读文字，而不是只依赖图标或空白态。

#### 4.2 还提供了更低门槛的 ID 直达模式

`assets/res/import/03/03fee81db.eea7c.json` 中可见：

- 输入框提示：`输入棋谱ID`
- 查询动作：`handler:"getQipuByID"`
- 返回节点：`backButtonNode`
- 返回文案：`返回`

这类“ID 直达”对无障碍用户很有价值：如果用户已经从群聊、分享页、老师口述或其他渠道拿到棋谱编号，就不必在长列表里逐项浏览，能够直接跳到目标内容。

#### 4.3 文字化状态反馈减少迷失感

从棋谱相关 prefab 能看到：

- `正在加载中...`
- `请稍后轻触重试`
- `暂无数据`
- `下拉刷新全部`
- `上拉加载更多`

这说明页面并没有把“等待、失败、空态、分页”藏在视觉细节里，而是尽量通过文案说明当前状态。对于依赖读屏或低视力浏览的用户，这类文字状态反馈非常重要。

### 5. 语音与棋步输入：不是自由听写，而是槽位式离散操作

#### 5.1 先教会用户怎么做

`assets/res/import/07/07ba69ace.1ec4d.json` 中出现：

- `长按发送语音`

这说明语音能力不仅存在，而且带有明显的**操作提示**。它没有假设用户天然知道语音入口在哪里，而是先告诉用户“怎么做”。

#### 5.2 语音下棋更像“可选槽位拼装”

`assets/res/import/09/097f406a0.6845d.json` 中可以直接看到：

- 按钮：`GO`
- 已选槽位文本：`红`、`前`、`五`、`进`、`四`

这组证据非常像把传统象棋口令拆成：

- 阵营/颜色
- 前后定位
- 棋子编号或列信息
- 动作
- 目标位

也就是说，它更可能不是把整句棋步完全交给自由语音识别，而是通过**离散槽位 + 最终确认**来构造棋步。这样做的好处是：

- 容错率更高
- 焦点遍历更清晰
- 不需要精确拖拽棋子
- 比完全依赖自然语言识别更稳定

#### 5.3 语音能力被接到局面编辑流里，而不是孤立功能

`assets/res/import/0c/0c1adc7f6.92ea0.json` 中可见：

- `VoiceOperationPanelLink`
- `VoiceOperationBingPanelLink`
- `摆满棋盘`
- `翻转`
- `红先`
- `完成`

这很关键，因为它说明语音/离散化操作并不只出现在聊天或提示层，而是已经深入到**创建局面**这种高复杂度流程里。复杂任务被拆成一组明确按钮，本身就是强无障碍信号。

### 6. 棋谱编辑与复杂任务：通过菜单化拆分步骤

`assets/res/import/05/053ecf2fb.5337c.json` 中可见：

- `菜单`
- `设置局面`
- `上一步`
- `下一步`

这组资源说明棋谱相关复杂任务不是依赖长手势、拖拽或隐藏操作，而是采用了**菜单项 + 顺序步骤**的形式。对于无障碍用户，这种设计比直接在棋盘上做高精度操作更友好，也更容易配合读屏逐项确认。

### 7. 分享与传播：二维码、固定平台选择、保存图片形成闭环

#### 7.1 “无障碍版”身份和二维码被直接做进分享条

`assets/res/import/0d/0ddee822c.7185a.json` 中可见：

- 文案：`天天象棋(无障碍版)`
- 资源名：`C_image_QR_code_01`
- 资源名：`Share_erweima_WX`

这意味着“无障碍版”并不是内部开关，而是会在分享物料上显式露出身份，同时带有二维码相关资源。对传播链路而言，这通常表示**扫码进入 / 快速传递 / 一键识别**被产品显式考虑过。

#### 7.2 分享目标不是开放式系统面板，而是固定可选项

`assets/res/import/0d/0dd04c445.97569.json` 中可见：

- `微信好友`
- `朋友圈`

这类固定目标面板的可用性价值在于：

- 用户不用先理解系统分享面板的复杂结构
- 焦点顺序稳定
- 平台命名直接可读
- 每个目标都可以做成大按钮

再结合前面的存储权限说明（保存图片用于分享），可以推断分享流程是被设计成完整闭环的，而不是“留给系统默认行为”。

### 8. 对这类无障碍实现方式的归纳

如果把这些证据合起来，这个 APK 的无障碍实现可以概括为：

1. **入口明确**：首页轻量、功能分类清晰。
2. **动作有限**：大量操作被收敛成按钮、列表项、输入框、固定面板。
3. **状态可读**：重试、空态、加载态、返回、标题、搜索提示都以文字出现。
4. **复杂任务被分步化**：棋谱编辑、局面创建、语音下棋都不是一次性完成，而是逐步确认。
5. **传播链路被产品化**：二维码、分享目标、保存图片等都被做成了显式交互。
